# AI 面试模块重构设计

日期：2026-04-23

## 背景

当前 AI 面试模块处于“逻辑已拆分、界面未收口”的半重构状态。`AiInterviewerPanel.vue` 已接入 `useInterviewSession`、`useInterviewTimer`、`useInterviewHint`、`useInterviewVoice`、`useInterviewHistory` 等组合式逻辑，但旧版完整的工作台结构、沉浸式模拟布局和阶段信息层没有被完整迁移，导致出现以下问题：

- 准备阶段右侧存在大面积空白，信息密度过低。
- AI 配置弹窗曾出现挂载失控、自动弹出、无法关闭的问题。
- 阶段切换、恢复会话、状态栏展示缺少统一壳层，容易出现局部状态残留。
- 页面视觉风格与交互层级不统一，模拟面试缺少“沉浸式主场景”的重心。

## 目标

本次重构的目标不是推翻现有逻辑，而是在保留当前 composable 和 service 调用链的前提下，恢复并升级旧版的完整界面结构，确保“准备 / 模拟 / 训练 / 复盘”四阶段都能稳定进入、退出和切换。

成功标准：

- AI 面试页面不再出现大面积无意义留白。
- 配置弹窗、简历预览、恢复会话弹窗都由父层受控挂载。
- 准备、模拟、训练、复盘四阶段都可正常进入和返回。
- 模拟阶段恢复沉浸式布局，计时、轮次、岗位、快捷操作集中展示。
- `npm run type-check` 和 `npm run build-only` 均通过。

## 设计方向

采用“恢复并升级旧版结构”的方案：

1. 非面试阶段使用旧版双栏工作台。
2. 模拟和训练阶段使用顶部状态栏加沉浸式工作区。
3. 保留现有逻辑层，不重写核心服务协议。

不采用“全部统一单布局”方案，因为当前模块已经有成熟的双栏与沉浸式两套结构资产，复用成本最低、稳定性最高。

## 结构设计

### 主壳层

`AiInterviewerPanel.vue` 重构为页面编排器，只负责：

- 当前阶段判定
- 顶部导航与阶段切换
- 工作台布局和沉浸式布局切换
- 全局状态栏
- 弹窗和浮层统一挂载

主壳层不再承担大量具体视图渲染细节。

### 阶段映射

| 主阶段 | 内部状态 | 主要组件 |
|---|---|---|
| 准备 | `jd-input` | `JdAnalysisStage` |
| 准备 | `prep-config` | `InterviewPrepPanel` |
| 模拟 | `simulation` | `InterviewSimulationPanel` |
| 训练 | `drill` | `InterviewDrillPanel` |
| 复盘 | `review` | `InterviewReviewPanel` |

### 布局策略

#### 准备阶段

恢复旧版 `workspace-layout`：

- 左侧主工作区显示 JD 输入、解析和面试前配置。
- 右侧侧栏显示岗位摘要、当前模式、时长、模型、操作引导。

#### 模拟 / 训练阶段

恢复旧版 `immersive-topbar + immersive-workspace`：

- 顶部固定显示计时、轮次、角色、岗位、快捷操作。
- 中间作为主训练区或主面试区。
- 右侧保留头像、状态、统计信息面板，并支持折叠。

#### 复盘阶段

复盘页回到工作台样式，但保留完整的信息块，而不是仅展示单张卡片。

## 模块适配策略

| 文件 | 处理方式 | 说明 |
|---|---|---|
| `AiInterviewerPanel.vue` | 大改 | 恢复旧版壳层并重接新逻辑 |
| `AiInterviewerPanel.old.vue` | 只读参考 | 提取稳定布局骨架和样式命名 |
| `InterviewSimulationPanel.vue` | 中改 | 适配沉浸式容器和提示区 |
| `InterviewPrepPanel.vue` | 中改 | 嵌入准备工作台并收口操作 |
| `InterviewDrillPanel.vue` | 中改 | 统一训练区层次与回退操作 |
| `InterviewReviewPanel.vue` | 中改 | 提升复盘层次与操作入口 |
| `AiConfigDialog.vue` | 小改 | 保持 `visible` 与 `close` 协议稳定 |
| `interview-tokens.css` | 小到中改 | 统一面试模块视觉 token |

## 状态流

### 进入条件

- 默认进入 `jd-input`
- 完成 JD 解析或复用历史分析后进入 `prep-config`
- 配置有效后可进入 `simulation` 或 `drill`
- 产生评估结果后进入 `review`

### 切换约束

- `simulation` 与 `drill` 需要有效岗位上下文。
- `review` 需要已有评估结果或已完成训练。
- `prep` 允许始终返回，作为主入口。

### 持久化与恢复

保留现有 localStorage / sessionStorage 会话恢复逻辑，但恢复入口统一由主壳层控制：

- 仅检测到有效会话快照时弹出恢复确认。
- 放弃恢复时同步清空本地缓存并回到初始流程。
- 恢复成功后同步恢复阶段、消息、JD 上下文、训练数据和复盘数据。

## 关键交互修正

### 配置弹窗

`AiConfigDialog` 只通过父层显式挂载，不允许因内部默认状态导致自动弹出。

### 顶部状态栏

计时、轮次、提示次数、目标岗位由主层统一控制显示时机，避免每个阶段各自维护。

### 简历预览

简历预览层继续保留为独立浮层，但挂载权归主壳层，避免模拟区或复盘区自行持有。

### 阶段切换

避免多个阶段组件“半存活”并同时留在 DOM 中，统一使用明确的条件渲染。

## 风险控制

本次重构遵循以下原则：

- 不修改核心服务协议和提示词调用链。
- 先重建壳层，再适配子面板，再修正交互细节。
- 优先复用旧版稳定结构，避免从零设计。
- 分批验证，确保任一阶段完成后页面仍可运行。

## 验证方案

### 功能链路

1. 准备链路：JD 输入、解析、复用历史、删除历史、进入配置。
2. 模拟链路：开始、发送文本、语音切换、请求提示、暂停、结束、复盘。
3. 训练链路：生成题目、回答、提交评估、返回上一阶段。
4. 复盘链路：查看评分、优化建议、跳转简历编辑、开始新一轮。
5. 全局链路：配置弹窗、恢复会话、简历预览、页面刷新稳定性。

### 构建验证

- `npm run type-check`
- `npm run build-only`

## 实施顺序

1. 重构 `AiInterviewerPanel.vue`，恢复旧版稳定壳层。
2. 将四个子面板重新挂到正确区域。
3. 收口 AI 配置弹窗、恢复弹窗、简历预览和状态栏。
4. 统一面试模块视觉 token 和局部样式。
5. 执行类型检查、构建和手动链路验证。
